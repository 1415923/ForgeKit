# 版本路线图

本文件只记录当前项目的版本规划，不记录 ForgeKit 模板自身的版本历史。

## 版本规划原则

- 大版本用于交付新的主要能力。
- 每个大版本结束后，安排一个 review/refactor/documentation 中版本。
- 如果 review/refactor gate 未完成，不应直接推进下一个大版本，除非用户明确接受风险。
- 未确认的内容写为 `待确认`，不要写成事实。

## 中版本 Review / Refactor 清单

每个 `x.y.1` 中版本至少完成：

- [ ] 梳理新增文件是否过多，是否可以合并、复用或抽象。
- [ ] 检查相似功能是否重复实现。
- [ ] 检查目录结构是否仍清晰。
- [ ] 检查模块边界是否被破坏。
- [ ] 检查接口、数据库、部署、测试文档是否同步。
- [ ] 检查测试是否覆盖核心路径。
- [ ] 检查配置、密钥、权限、安全边界。
- [ ] 更新 `docs/技术债记录.md`。
- [ ] 更新 `docs/质量指标记录.md`。
- [ ] 高影响变更已更新 `docs/变更影响评估.md`。
- [ ] 更新 `docs/架构设计.md`、`docs/版本更新记录.md`。
- [ ] 给出是否允许进入下一个大版本的结论。

| --- | --- | --- |
