# 测试规则

## 测试策略

- 小改动至少运行相关测试。
- 共享模块、接口、数据结构变化需要运行更大范围测试。
- Bug 修复应补充回归测试，除非项目条件不允许。
- 无法运行测试时，最终说明原因和替代验证方式。
- 高风险核心逻辑优先采用 TDD：先写失败测试，再实现，再重构。
- 覆盖率目标按项目确定；80% 可作为中大型业务项目的推荐目标，不作为默认硬性要求。

## 验证闭环

一次完整验证可以包含：

1. 单元测试。
2. 集成测试。
3. 端到端测试。
4. 类型检查或静态检查。
5. 构建。
6. 安全检查。

根据变更风险选择必要项，不机械运行无关命令。

## 测试类型

| 类型 | 工具 | 命令 | 适用场景 |
| --- | --- | --- | --- |
| 单元测试 | 待补充 | 待补充 | 核心逻辑 |
| 集成测试 | 待补充 | 待补充 | 模块协作 |
| 端到端测试 | 待补充 | 待补充 | 用户主流程 |
| 静态检查 | 待补充 | 待补充 | 提交前 |
| 安全检查 | 待补充 | 待补充 | 依赖、凭据、权限相关变更 |

## 覆盖重点

- 核心业务规则。
- 权限和鉴权。
- 数据输入校验。
- 边界条件和异常路径。
- 接口兼容性。

## 验证记录

每次较大变更后，在最终回复或版本记录中说明：

- 运行了哪些命令。
- 结果如何。
- 哪些测试未运行。
