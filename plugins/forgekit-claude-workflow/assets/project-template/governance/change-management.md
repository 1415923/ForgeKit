# 变更管理流程

本文件定义项目中高影响改动的评估、确认、实现、验证和回滚规则。

## 何时必须做变更影响评估

以下改动必须先填写 `docs/变更影响评估.md`：

- API 路径、请求参数、响应结构、错误码变化。
- 数据库表结构、字段、索引、迁移、数据语义变化。
- 配置项、环境变量、端口、启动方式变化。
- 部署方式、构建产物、运行环境变化。
- 鉴权、授权、权限、Token、Session、Cookie 变化。
- 用户数据、敏感数据和密钥处理变化。
- 文件上传、下载、存储路径、对象存储结构变化。
- 外部服务、中间件、模型服务、第三方 API 变化。
- 硬件接口、FPGA bitstream、寄存器映射、板卡连接方式变化。
- 会影响上下游系统、客户使用方式、生产数据或回滚方式的变化。
- 命中 Critical 或 Unknown 代码所有权区域的变化。

## 风险等级

| 等级 | 说明 | 要求 |
| --- | --- | --- |
| 高 | 影响外部接口、数据、安全、部署、硬件或生产行为 | 必须人工确认，必须有回滚或规避方案 |
| 中 | 影响内部模块、配置、测试或局部兼容 | 需要评估和验证 |
| 低 | 局部实现细节，不影响外部行为 | 记录即可 |

## 接手项目特殊规则

接手既有项目时，默认把以下变更视为高风险：

- 对外 API 改动。
- 数据库结构或语义改动。
- 鉴权和权限改动。
- 部署配置改动。
- 上下游服务契约改动。

没有测试、文档和人工确认时，不应直接执行高风险变更。

## 变更流程

```text
识别变更 -> 影响评估 -> 风险分级 -> 人工确认 -> 实现 -> 验证 -> 文档同步 -> 版本记录 -> 回滚准备
```

## Codex 行为

- 发现高影响变更时，先提醒填写 `docs/变更影响评估.md`。
- 如果缺少影响评估，不应直接修改接口、数据库、权限、部署或硬件接口。
- 如果用户要求跳过评估，必须记录风险并要求明确确认。
- 发布检查必须确认所有高/中风险变更已有评估和验证。
- 安全敏感变更还必须参考 `governance/security-governance.md`。
- Critical 或 Unknown 代码区域变更还必须参考 `governance/code-ownership-review-governance.md`。
