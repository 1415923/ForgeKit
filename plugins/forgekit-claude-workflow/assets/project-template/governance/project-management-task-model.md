# 项目管理任务模型

本文件定义 Epic、Feature、Task、Bug 的层级、状态流和版本关系。目标是让 Codex 在规划、实现、审查和发布时能把工作拆清楚，而不是只根据一句需求直接写代码。

## 层级模型

```text
Epic
  -> Feature
      -> Task
      -> Bug
      -> Test
```

| 类型 | 前缀 | 粒度 | 说明 |
| --- | --- | --- | --- |
| Epic | `EPIC` | 一个阶段或业务主题 | 通常跨多个版本或多个 Feature |
| Feature | `FEAT` | 一个可验收能力 | 应能关联需求、验收标准和版本 |
| Task | `TASK` | 可执行工作项 | 应能在一个短周期内完成 |
| Bug | `BUG` | 缺陷修复项 | 应有复现、严重程度和验证方式 |
| Test | `TEST` | 验证项 | 应关联需求、Feature、Task 或 Bug |

## 状态流

推荐状态：

```text
Backlog -> Ready -> In Progress -> Review -> Done
                 -> Blocked
                 -> Dropped
```

| 状态 | 含义 | 进入条件 |
| --- | --- | --- |
| Backlog | 已记录，未准备开发 | 有基本标题和背景 |
| Ready | 可以进入开发 | 满足 Definition of Ready |
| In Progress | 正在实现 | 已确认范围和负责人 |
| Review | 等待审查或验收 | 实现完成，验证材料可用 |
| Done | 已完成 | 满足 Definition of Done |
| Blocked | 被阻塞 | 阻塞原因和下一步明确 |
| Dropped | 放弃或裁剪 | 记录原因和影响 |

## 版本关系

- 每个 Feature、Task、Bug 应归属到一个目标版本或明确为 Backlog。
- 每个大版本应只包含能形成闭环的 Feature，不把所有想法都塞进当前版本。
- Review/refactor 中版本应包含技术债、重构、文档同步、测试补齐和质量改进任务。
- 未完成任务必须明确是延期、裁剪、阻塞还是放弃。

## 拆分规则

一个 Task 应满足：

- 范围清楚。
- 输入输出清楚。
- 验证方式清楚。
- 影响文件或模块大致清楚。
- 不混合多个无关目标。

如果一个 Task 同时修改 API、数据库、前端、部署和权限，通常应拆成多个 Task，并用 Feature 或 Epic 关联。

## Bug 严重程度

| 等级 | 说明 | 默认处理 |
| --- | --- | --- |
| P0 | 系统不可用、数据损坏、安全严重问题、主流程中断 | 立即处理，不推进新功能 |
| P1 | 重要功能错误、兼容风险、重要测试缺口 | 当前版本优先处理 |
| P2 | 局部问题、质量问题、可规避缺陷 | 进入计划或 review/refactor |
| P3 | 优化、建议、长期改进 | 进入 Backlog |

## Codex 行为

- 开始开发前，确认当前工作对应 EPIC/FEAT/TASK/BUG 编号。
- 如果用户只给出模糊目标，先拆 Feature 和 Task，再确认优先级。
- 不满足 Ready 的工作不直接进入编码。
- 完成工作时更新任务状态、追踪矩阵和版本记录。
- 发现任务过大或混合多个目标时，建议拆分。
- 发现 P0/P1 Bug 时，优先稳定项目，不建议继续新增大功能。
