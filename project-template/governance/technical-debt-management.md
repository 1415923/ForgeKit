# 技术债管理流程

技术债是当前接受、但未来必须跟踪或偿还的设计、实现、测试、文档或运维问题。

## 什么时候记录技术债

- 为了赶进度临时实现。
- 接手项目中发现重复代码、结构混乱、测试缺失。
- 大版本中暂时不处理的架构问题。
- 跳过 review/refactor 中版本的部分检查。
- 已知性能、安全、兼容或部署问题暂时接受。
- 代码实现与 ADR/RFC 的理想方案有偏差。
- 为了兼容或短期交付，暂缓高影响变更中的完整修复。
- 事故或重复缺陷暴露出的系统性问题。

## 编号规则

```text
DEBT-001
DEBT-002
```

## 分类

| 类型 | 示例 |
| --- | --- |
| 架构债 | 模块边界不清、依赖方向混乱 |
| 代码债 | 重复实现、临时分支、过长函数 |
| 测试债 | 核心路径没有测试、只能手工验证 |
| 文档债 | 接口、部署、架构文档不同步 |
| 安全债 | 权限校验不完整、依赖审查缺失 |
| 运维债 | 启动步骤复杂、环境变量混乱 |

## 处理状态

| 状态 | 含义 |
| --- | --- |
| Open | 已记录，未处理 |
| Planned | 已排入版本 |
| In Progress | 正在处理 |
| Paid Down | 已偿还 |
| Accepted | 明确接受，不再处理 |

## Codex 行为

- 发现不能立即处理的问题时，不要隐藏，应记录到 `.forgekit/docs/technical-debt.md`。
- 大版本结束后的 review/refactor 中版本必须检查技术债。
- 如果技术债影响版本推进，应在 `.forgekit/docs/version-roadmap.md` 和 `.forgekit/docs/risk-register.md` 中体现。
- 如果技术债来自高影响变更，应关联 `.forgekit/docs/change-impact.md`。
- 如果技术债来自事故或缺陷复盘，应关联 `.forgekit/docs/incident-review.md` 或 `.forgekit/docs/defect-review.md`。
- 接手项目时先建立技术债基线，再逐步偿还。
