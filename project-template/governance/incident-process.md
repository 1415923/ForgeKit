# 事故 / 缺陷复盘流程

本文件定义生产事故、严重缺陷、发布失败、数据问题、安全问题和重大回归的复盘流程。

## 什么时候必须复盘

以下情况必须创建复盘记录：

- 生产或准生产服务不可用。
- 发布失败、回滚或热修。
- 数据损坏、数据丢失、错误写入。
- 安全漏洞、密钥泄露、权限绕过。
- 核心业务流程不可用。
- 严重兼容性破坏。
- 同类 Bug 重复出现。
- 接手项目中发现历史遗留的高影响缺陷。

## 复盘原则

- 复盘目标是改进系统，不是追责。
- 先记录事实，再分析原因。
- 区分直接原因、根因、促成因素。
- 每个行动项必须有编号、负责人、目标版本和验证方式。
- 行动项应同步到风险、技术债、测试、版本路线图。

## 流程

```text
事件识别 -> 临时止血 -> 影响评估 -> 根因分析 -> 修复方案 -> 行动项 -> 验证 -> 关闭
```

## 严重程度

| 等级 | 说明 | 要求 |
| --- | --- | --- |
| SEV-1 | 数据损坏、安全事故、核心服务不可用 | 必须复盘，阻塞后续发布 |
| SEV-2 | 核心流程严重异常、发布回滚、重要客户受影响 | 必须复盘 |
| SEV-3 | 局部功能异常、有规避方案 | 建议复盘 |
| SEV-4 | 低影响问题 | 可记录到缺陷修复计划 |

## Codex 行为

- 遇到 SEV-1 / SEV-2，不应只给修复代码，应要求创建复盘记录。
- 修复后应检查是否需要新增测试、风险项、技术债、变更影响评估或 ADR。
- 安全事故应同步到 `docs/threat-model.md` 和 `docs/risk-register.md`。
- 发布检查应确认严重事故行动项是否关闭或被明确接受。
