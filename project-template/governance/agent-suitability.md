# AI Agent 项目适用性评估

本文用于判断项目是否适合直接使用 Codex agent 工作流。结论不是“能不能用 AI”，而是判断应直接套用模板、先补基础工程条件，还是需要定制化流程。

## 评估结论

| 结论 | 含义 | 推荐动作 |
| --- | --- | --- |
| Suitable | 可以直接使用模板 | 按 Lite / Standard / Enterprise 初始化 |
| Conditional | 可以使用，但要先补条件 | 先补 Git、目录、测试、文档或工具链 |
| Custom | 不适合直接套模板 | 定制流程，减少自动化和大范围修改 |

## 核心问题

| 问题 | Suitable 信号 | Conditional / Custom 信号 |
| --- | --- | --- |
| 是否 Git 仓库 | 有清晰历史、分支、diff | 无 Git、非标准 VCS、历史不可追踪 |
| 是否标准工程目录 | 有清晰源码、测试、配置、文档入口 | 目录混乱、构建产物和源码混杂 |
| 是否主要是文本代码 | 主要是代码、配置、文档 | 大量二进制、图片、模型、工程产物 |
| 是否有可运行验证 | 有测试、构建、lint 或启动命令 | 无法验证、命令缺失、环境不可复现 |
| 是否有业务/架构说明 | 有 README、接口、数据、部署说明 | 只能靠口头经验或隐性知识 |
| 是否工程师主导维护 | 有技术负责人和评审机制 | 非工程团队主导，变更不可审查 |

## 不适合直接套模板的情况

- 无 Git 或历史不可追踪。
- 主要资产是二进制、大型设计文件、模型权重、图片或办公文档。
- 不能本地构建、测试、启动，也没有可替代验证方式。
- 项目依赖大量人工 GUI 操作且无脚本化入口。
- 变更必须由非工程系统、供应商工具或线下流程决定。
- 安全、合规、生产风险极高，但没有负责人和审批链。

## 处理策略

| 问题 | 先做什么 |
| --- | --- |
| 无 Git | 先初始化仓库或建立只读快照，不做大改 |
| 目录混乱 | 先写 `docs/codebase-map.md` 和接手审计 |
| 无验证命令 | 先补 `docs/local-toolchain.md`，记录最低验证方式 |
| 大量二进制 | 只记录元数据和生成方式，不读入上下文 |
| 架构不清 | 先走探索报告和实施计划 |
| 非工程师主导 | 明确人工确认点和不可自动化边界 |

## 编码前闸门

如果评估结论是 Conditional 或 Custom，Codex 不应直接开始大范围编码。必须先输出：

- 缺失条件。
- 可补齐条件。
- 需要人工确认的问题。
- 可安全执行的最小下一步。
- 是否允许进入编码。
