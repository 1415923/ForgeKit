# v0.3 Agent Harness 路线图

本路线图用于把“大代码库 AI agent 工程实践”的 7 个问题转化为本模板的后续版本计划。核心判断：模型能力只是下限，真正决定工程效果的是 harness，也就是规则入口、搜索方式、上下文分层、skills、hooks、LSP、MCP、团队分发和维护机制。

## 总目标

v0.3 系列目标不是继续堆文档，而是让模板更像成熟 AI agent 工程 harness：

- Codex 有清晰、短小、分层的入口。
- Codex 知道从哪里开始搜代码，而不是全项目乱翻。
- 大任务能先探索、写计划、再拆会话执行。
- 技术栈模板提供 LSP、局部测试、子目录启动建议。
- 高频流程沉淀为 skill / command / hook 候选。
- 团队可以把模板、skills、工具链作为组织资产维护。
- 项目初始化时能判断项目是否适合 AI agent 工作流。

## 文章 7 问到模板能力映射

| 文章问题 | 通用工程含义 | 模板吸收方向 |
| --- | --- | --- |
| Q1 context 爆 | 不能全量读代码，必须给 agent 好起点 | AGENTS 分层、代码库地图、agentic search 指南 |
| Q2 规则文件过长 | 入口规则必须短，细节下沉 | AGENTS 长度控制、子目录 AGENTS、周期审查 |
| Q3 找错函数 | 只靠文本搜索不够 | LSP、子目录启动、局部测试命令、ignore 规则 |
| Q4 大改崩溃 | 跨文件任务要拆会话和计划 | 大任务执行协议、探索报告、分阶段实现 |
| Q5 团队推广 | 好实践要 skill/plugin/MCP 化 | skill 分层、plugin 化规划、团队 DRI |
| Q6 创始人用法 | 权限、Plan Mode、hooks、常用命令 | 权限白名单、hook 示例、重复任务转 skill |
| Q7 不适合项目 | AI agent 有适用边界 | 项目适用性评估 |

## 版本拆分

### v0.3.0 Agent Harness 基线

目标：

- 建立本模板的 agent harness 总论。
- 明确 AGENTS.md 的定位、长度、分层规则和审查周期。
- 新增代码库地图模板，让 Codex 有搜索起点。
- 在 HTML 工作台中输出“推荐启动目录”和“需要先读的入口”。

具体任务：

- 新增 `governance/agent-harness.md`。
- 新增 `docs/代码库地图.md`。
- 更新 `AGENTS.md` 和 `project-template/AGENTS.md`，增加短入口和分层规则。
- 更新 `scripts/validate-template.ps1`，检查 AGENTS 是否存在、是否过长、是否引用核心入口。
- 更新 `使用说明.html`，根据技术栈和入口输出推荐启动目录。
- 更新 README，说明 harness 优先级：AGENTS -> skill -> stack -> governance details。

验收效果：

- 新项目复制后，Codex 先读 `AGENTS.md`，再按任务加载相关文档。
- 不再默认让 Codex 阅读全部治理文档。
- 项目有一个可维护的代码库地图入口。

### v0.3.1 Harness Review / Refactor Gate

目标：

- 对 v0.3.0 做 review/refactor 中版本。
- 检查 AGENTS、代码库地图、HTML、skills 是否重复或冲突。
- 修复 v0.3.0 引入的文档漂移。

具体任务：

- 更新 `docs/质量指标记录.md`，记录上下文入口质量。
- 更新 `docs/技术债记录.md`，记录 harness 债务。
- 检查所有 skills 是否仍保持 ASCII、短小、按需加载。
- 检查 `使用说明.html` 是否仍然用户友好。
- 更新 `scripts/validate-template.ps1` 的自检项。

验收效果：

- v0.3.0 新增内容不造成上下文膨胀。
- 所有新增入口都有明确用途，没有重复提示词。

### v0.4.0 LSP / 子目录启动 / 局部验证

状态：已完成第一版

目标：

- 把“找错文件、跑错测试”的问题纳入技术栈模板。
- 让不同语言项目都有推荐的符号搜索、启动目录和局部验证命令。

具体任务：

- 更新 `templates/java-springboot/`：Java LSP / IDEA / Maven 局部测试建议。
- 更新 `templates/vue/`、`templates/react/`：TypeScript LSP、前端局部 lint/build/test。
- 更新 `templates/python-fastapi/`：Pyright / pytest 局部测试。
- 更新 `templates/node-express/`：TypeScript / ESLint / npm scripts。
- 更新 `templates/fpga-vivado-vitis/`：Vivado/Vitis 工程目录、仿真、综合、HLS 局部验证。
- 新增 `docs/本地工具链检查.md` 或扩展环境矩阵，记录 LSP / lint / test 可用性。

验收效果：

- Codex 能根据技术栈知道从哪个子目录启动、用什么命令验证。
- 大项目中减少全局搜索和全量测试。

### v0.7.0 项目适用性评估和真实项目回灌

状态：已完成第一版

目标：

- 明确哪些项目适合 AI agent 工作流，哪些需要定制化。
- 推进 MAT-005 真实项目回灌。

具体任务：

- 新增 `governance/agent-suitability.md`。
- 新增 `docs/项目适用性评估.md`。
- HTML 工作台增加适用性问题：
  - 是否 Git 仓库。
  - 是否标准工程目录。
  - 是否主要是文本代码。
  - 是否有可运行测试或构建。
  - 是否大量依赖二进制资源。
  - 是否非工程师主导维护。
- 新增 `docs/真实项目试用记录.md`。

验收效果：

- 初始化阶段能判断项目是否适合直接套用模板。
- 对不适合的项目，提示需要定制化，而不是硬套 Standard / Enterprise。
