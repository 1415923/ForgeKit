# 版本治理

本文件定义版本规划、发布、review/refactor 中版本和推进闸门。

## 版本类型

| 类型 | 示例 | 目的 |
| --- | --- | --- |
| 功能大版本 | `v0.1.0`、`v0.2.0` | 交付新的主要能力 |
| Review/Refactor 中版本 | `v0.1.1`、`v0.2.1` | 整体 review、复用整理、文档同步、测试补齐 |
| 修复版本 | `v0.1.2` | 修 bug、补兼容、修安全 |
| 发布版本 | `v1.0.0` | 对外稳定交付 |

## 推进规则

- 功能大版本完成后，不直接进入下一个功能大版本。
- 必须先完成对应 review/refactor 中版本。
- 中版本应减少新增大功能，重点整理结构和技术债。
- 如果跳过中版本，必须人工确认并记录原因。

## Review/Refactor 中版本检查

- 重复逻辑是否抽取。
- 文件数量是否异常增长。
- 模块边界是否清楚。
- 接口和数据库文档是否同步。
- 环境矩阵、发布流水线和部署文档是否同步。
- Critical 或 Unknown 代码区域是否已有评审结论。
- 测试是否覆盖核心路径。
- 安全和权限是否有新风险。
- S2/S3 安全风险是否关闭或被明确接受。
- 是否需要新增 ADR。
- 技术债是否记录、偿还或接受。
- 质量指标是否更新。
- 严重事故和重复缺陷是否复盘。

## 与发布检查的关系

发布检查回答：当前版本能否发布。

版本闸门回答：能否进入下一个大版本。

两者都通过，才建议继续推进。

## 与技术债和质量指标的关系

- 大版本结束后，必须更新 `docs/technical-debt.md` 和 `docs/quality-metrics.md`。
- 如果高风险技术债未处理，默认不建议进入下一个大版本。
- 如果质量指标明显恶化，应安排 review/refactor 中版本。
- 如果存在未关闭 SEV-1 / SEV-2 行动项，默认不建议进入下一个大版本。
- 如果存在未关闭 S2/S3 安全风险，默认不建议发布或进入下一个大版本。
- 如果发布流水线、环境矩阵或回滚方案缺失，默认不建议生产发布。
- 如果 Critical 或 Unknown 代码区域缺少评审结论，默认不建议发布或进入下一个大版本。
