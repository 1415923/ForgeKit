# 版本路线图

本文件定义项目的大版本规划、中版本 review/refactor 闸门，以及每个版本的推进条件。

## 版本规划原则

- 大版本用于交付新的主要能力。
- 每个大版本结束后，必须安排一个中版本用于整体 review、重构、文档同步和架构梳理。
- 如果 review/refactor 中版本未完成，不应推进下一个大版本。
- 不允许每个版本只追加文件和功能，而不复用已有能力、不清理架构债务。

## 推荐版本节奏

```text
v0.1.0 功能闭环 1
v0.1.1 review/refactor/documentation gate
v0.2.0 功能闭环 2
v0.2.1 review/refactor/documentation gate
v0.3.0 功能闭环 3
v0.3.1 review/refactor/documentation gate
```

## 中版本 Review / Refactor 清单

每个 `x.y.1` 中版本至少完成：

- [ ] 梳理新增文件是否过多，是否可以合并、复用或抽象。
- [ ] 检查相似功能是否重复实现。
- [ ] 检查目录结构是否仍清晰。
- [ ] 检查模块边界是否被破坏。
- [ ] 检查接口、数据库、部署、测试文档是否同步。
- [ ] 检查测试是否覆盖核心路径。
- [ ] 检查配置、密钥、权限、安全边界。
- [ ] 更新 `docs/技术债记录.md`。
- [ ] 更新 `docs/质量指标记录.md`。
- [ ] 高影响变更已更新 `docs/变更影响评估.md`。
- [ ] 更新 `docs/架构设计.md`、`docs/版本更新记录.md`。
- [ ] 给出是否允许进入下一个大版本的结论。
