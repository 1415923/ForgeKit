本文件用于接手已有项目时的第一阶段工作。目标是先理解现状、识别风险、修复明确 bug，而不是立即重构架构。

## 3. 兼容边界

接手项目时，默认不改变以下内容，除非人工确认：

- 对外 API 路径、请求、响应、错误码。
- 数据库表结构和已有数据语义。
- 上下游服务调用协议。
- 鉴权、权限、Token、Cookie、Session 机制。
- 部署方式、端口、环境变量。
- 文件存储路径和外部对象存储结构。
- 关键业务流程和用户可见行为。

## 5. 修复优先级

| 优先级 | 类型 | 处理原则 |
| --- | --- | --- |
| P0 | 会导致无法启动、数据损坏、安全高危、主流程不可用 | 立即修复 |
| P1 | 明确 bug、兼容风险、重要测试缺口 | 当前接手阶段处理 |
| P2 | 代码质量、重复逻辑、局部设计问题 | 进入 review/refactor 中版本 |
| P3 | 大架构调整、技术栈替换、深层重构 | 进入后续版本规划，不能直接动 |

## 6. 接手阶段结论

- 是否能稳定启动：否 / 是
- 是否能运行核心流程：否 / 是
- 是否存在阻塞交付的问题：待补充
- 是否允许进入后续功能开发：否 / 是
- 需要人工确认的兼容风险：待补充

兼容风险和高影响风险应同步到 `docs/风险登记册.md`。
