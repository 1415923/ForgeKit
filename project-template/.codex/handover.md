# 既有项目接手规则

用于接手已有项目、修 bug、再做后续开发的场景。

## 阶段顺序

1. 现状审计。
2. 大规模代码审查。
3. P0/P1 缺陷修复。
4. 兼容边界确认。
5. 补齐文档、测试和版本记录。
6. 基于现有技术栈、实际需求和落地条件，制定后续开发路线图。

## 禁止默认行为

接手阶段默认禁止：

- 大规模重构。
- 替换技术栈。
- 改变对外接口。
- 改变数据库语义。
- 改变部署方式。
- 清理或删除看似无用但未确认的代码。
- 未填写变更影响评估就修改 API、数据库、权限、部署或上下游契约。

## 允许行为

接手阶段优先允许：

- 只读分析。
- 运行测试和构建。
- 复现 bug。
- 修复明确问题。
- 增加回归测试。
- 补充文档。
- 小范围整理，不改变外部行为。

## 推进条件

只有满足以下条件，才建议进入后续功能开发：

- 核心流程可运行。
- P0 已修复或规避。
- P1 已修复或明确排期。
- 兼容边界已记录。
- 后续版本路线图已更新。
- 人工确认允许进入下一阶段。

版本规划和版本推进闸门仍然适用。

接手项目中的高影响变更还必须参考 `governance/change-management.md`，并填写 `docs/change-impact.md`。
