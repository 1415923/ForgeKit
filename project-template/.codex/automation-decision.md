# 自动化决策矩阵

本文用于判断一类能力应该做成 skill、command、hook、script、MCP，还是暂时只保留文档。

## 判断顺序

| 层级 | 适合放什么 | 不适合放什么 |
| --- | --- | --- |
| AGENTS / CLAUDE | 入口、路由、安全边界、少量强约束 | 长流程、技术栈手册、重复清单 |
| Skill | 需要 AI 判断、访谈、阅读证据、产出方案或审查意见的稳定工作流 | 固定命令、简单脚本、纯机械检查 |
| Command | 可手动触发、低风险、可重复的固定动作或提示词 | 高频必须自动提醒的检查 |
| Hook | 低风险、可解释、可跳过、失败信息清楚的自动检查 | 会联网、写外部系统、安装依赖、部署、push、tag、迁移 |
| Script | 只读检测、跨平台本地检查、固定格式校验 | 需要复杂上下文判断的方案决策 |
| MCP | 需要外部系统上下文或动作，且权限、凭据、审计可控 | 本地文档/代码能解决的问题 |
| 文档 | 低频、未稳定、需要人工判断的流程 | 已经高频重复且遗漏成本高的流程 |

## 适合做 Hook

- 文档同步提醒，例如版本记录、相关文档、过期描述。
- 提交前只读格式、结构、入口文件检查。
- 大任务前提醒探索报告和实施计划是否存在。
- 发布前提醒版本记录、测试记录、任务状态是否齐。
- 安全敏感路径变更时提醒安全审查。

默认从 warning 模式开始，确认噪音可接受后再 strict。

## 不建议做 Hook

- 自动改文档或自动生成版本记录。
- 自动安装依赖、初始化工具、启动长期服务。
- 自动提交、push、tag、部署、数据库迁移。
- 需要联网、凭据、issue/PR 写入或外部系统状态变更的动作。
- 需要产品、架构或业务取舍的决策。
- 在小项目里会大量误报的 Enterprise 级检查。

## 适合做 Skill

- `project-init`：初始化访谈和路线确认。
- `project-bootstrap-fill`：根据问答生成第一版项目文档。
- `project-suitability`：判断项目是否适合 ForgeKit 工作流。
- `document-backfill`：逐篇消化既有文档并回填项目事实。
- `handover-review`：接手项目审计。
- `large-change-planning`：大任务探索和分阶段计划。
- `code-review`：基于 diff 或范围的风险审查。
- `release-check`：发布闸门审查。
- `security-review`：安全敏感变更审查。

## 暂不做成内置 Skill

- `api-design`：不同项目风格差异大，先按项目自建。
- `e2e-testing`：依赖产品路径和测试框架，先按项目自建。
- `docs-research`：更像通用联网研究能力，不应默认内置到模板。
- 单一语言/技术栈规则：继续放在 `.codex/stacks/`。
- 本地检测和文档同步：保留为 script + optional hook。

## 当前推荐 Hook

| Profile | 目标 | 默认 |
| --- | --- | --- |
| `docs-warn` | 文档同步提醒，不阻断提交 | 推荐 |
| `docs-strict` | 文档同步检查失败时阻断提交 | 团队确认后启用 |

启用方式见 `.codex/hooks.md`。
